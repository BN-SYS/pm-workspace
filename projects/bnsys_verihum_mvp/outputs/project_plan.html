Section 1

서비스 개요

1.1 서비스 정의

  • 서비스명: Verihum (Verify + Human)
  • 분류: AI 기반 사람 작성 인증 SaaS 플랫폼
  • 핵심 콘셉트: 결과물이 아닌 작성 과정 데이터를 수집·분석하여 사람이 직접 작성했는지 여부를 판별하고 인증

1.2 핵심 가치 제안

"학생은 평소처럼 과제를 작성합니다. 제출 후 교사는 작성 과정 리포트로 AI 활용 여부를 확인합니다."

1.3 MVP 개발 목표

  • 교사: 과제 등록 → 제출 현황 확인 → 리포트 열람
  • 학생: 과제 확인 → Verihum 에디터에서 작성 → 제출
  • 시스템: 작성 과정 자동 수집 → AI 분석 → 리포트 자동 생성

1.4 서비스 철학: 올바른 AI 활용

"Verihum은 AI 사용을 금지하는 서비스가 아닙니다. AI를 올바르게 활용하는 능력을 기르는 서비스입니다."

Verihum은 학생이 AI를 활용하는 행위 자체를 부정적으로 보지 않습니다. 오히려 AI에게 좋은 질문을 던지고 다양한 관점의 답변을 얻는 것은 학습 역량을 키우는 긍정적인 행동입니다.

문제는 AI가 제시한 답변을 비판적 사고 없이 그대로 수용하는 것입니다. Verihum은 학생이 AI 제안을 어떻게 검토하고, 수정하고, 자신의 언어로 재해석했는지 — 즉 사고의 과정을 측정합니다.

행동 유형 평가 관점 비고
AI에게 질문하고 답변을 참고함 긍정적 활용 AI를 도구로 활용하는 능동적 학습
AI 제안을 검토·수정 후 자신의 언어로 재작성 올바른 활용 비판적 사고 과정이 기록됨
AI 제안을 수정 없이 그대로 붙여넣기 주의 필요 사고 과정 없이 결과물만 수용한 경우
AI 제안을 거부하고 직접 작성 자기주도 작성 독립적 사고력 발현
핵심 평가 기준: "AI를 썼는가"가 아니라 "AI를 어떻게 활용했는가"가 Verihum 인증 리포트의 핵심 판단 기준입니다. AI 제안을 받아들인 비율보다, 제안을 얼마나 비판적으로 검토하고 자신의 사고로 전환했는지가 평가에 반영됩니다.
Section 2

사용자 역할 정의

교사 (Teacher) MVP포함

  • 역할코드: TEACHER
  • 주요 행위: 과제 등록, 학생 초대/관리, 제출 현황 확인, 리포트 열람, 과정 리플레이 확인
  • 접근 권한: 자신이 등록한 과제 및 제출물 전체 조회 권한

학생 (Student) MVP포함

  • 역할코드: STUDENT
  • 주요 행위: 과제 확인, 에디터에서 글 작성, 과제 제출, 본인 결과 확인
  • 접근 권한: 자신에게 배정된 과제 및 본인 제출물로 제한

관리자 (Admin) MVP포함

  • 역할코드: ADMIN
  • 주요 행위: 교사 계정 등록 및 관리, 교사별 과제·학생 현황 조회, 전체 서비스 운영 모니터링
  • 접근 권한: 전체 교사·학생·과제·제출물 데이터 조회 권한
  • Note: MVP는 BN System 내부 운영자 전용 기본 기능 한정. 통계 대시보드 및 요금제 관리는 차기
Section 3

화면 구성 및 기능 정의

3-1. 인증 화면 (AUTH)

로그인

SCR-AUTH-01 MVP포함
  • 기능: 이메일 + 비밀번호 로그인
  • 입력값: 이메일, 비밀번호
  • 출력 / 분기: 역할에 따라 분기 — TEACHER → 과제 목록 / STUDENT → 내 과제 목록
  • 정책: 로그인 실패 시 에러 메시지 표시. 계정 잠금 정책은 MVP 범위 외
  • 협의 필요: 소셜 로그인(Google Workspace) 포함 여부 → OI-02 연계

회원가입

SCR-AUTH-02 MVP포함
  • 기능: 이메일 기반 신규 회원가입
  • 입력값: 이름, 이메일, 비밀번호, 역할 선택(교사/학생)
  • 정책: 이메일 중복 확인 필수. 이메일 인증 절차는 MVP 범위 외(OI-01)
  • Note: 교사가 직접 학생 초대하는 방식도 검토 필요 (OI-03 참고)

3-2. 교사 화면 (TEACHER)

과제 목록

SCR-TCHR-01 MVP포함
  • 표시 정보: 과제명, 과목, 제출 기간, 제출 현황 (n명 제출 / 전체), 상태 (진행중 / 종료 / 임시저장)
  • 기능: 과제 등록 버튼, 과제 클릭 시 상세 이동
  • 정렬: 최신 등록순 기본

과제 등록

SCR-TCHR-02 MVP포함
  • 입력 필드:
    • 과제명 (필수, 최대 100자)
    • 과목 (선택형 또는 직접 입력 — OI 협의)
    • 과제 안내문 (선택, 텍스트)
    • 제출 기간 (시작일시 ~ 종료일시)
    • 학생 배정 방식: 링크 공유 or 이메일 초대 (OI-03 — MVP는 링크 공유 방식 권장)
  • 등록 완료 시: 학생 공유 링크 자동 발급
  • 협의 필요: 과제 수정/삭제 정책 (OI-04)

과제 상세 / 제출 현황

SCR-TCHR-03 MVP포함
  • 표시 정보: 과제 기본 정보, 학생별 제출 상태 테이블
  • 테이블 컬럼: 학생명 / 제출일시 / AI 분석 상태(분석 중 / 완료) / 인증 결과 / 리포트 열람 버튼
  • 기능: 리포트 열람 버튼 클릭 → SCR-TCHR-04 이동

개별 리포트 열람

SCR-TCHR-04 MVP포함
  • 표시 정보:
    • 학생명, 과제명, 제출일시
    • 인증 결과: 직접 작성 인증 / 주의 필요 / 분석 불가 (3단계 — OI-09 확정 필요)
    • 과정 점수 (0 ~ 100)
    • 주요 분석 수치 요약 (제너럴한 표현 한정)
    • 과정 리플레이 버튼
  • 정책: 리포트는 교사만 열람 가능. 학생 공유 여부 협의 (OI-10)
특허 보호: 기술 세부 수치(타이핑 속도, 키 입력 패턴 등) 화면 노출 금지

과정 리플레이

SCR-TCHR-05 협의필요
  • 기능: 학생의 과제 작성 과정을 타임라인 형태로 재현
  • 컨트롤: 재생/일시정지, 배속 조절(1x / 2x), 타임라인 슬라이더
협의 필요 (OI-11): 리플레이 구현 방식(스냅샷 기반 vs 이벤트 스트리밍)에 따라 공수 차이가 크므로 개발팀 협의 우선

3-4. 관리자 화면 (ADMIN)

교사 목록 및 등록

SCR-ADMIN-01 MVP포함
  • 표시 정보: 등록된 교사 목록 (이름, 이메일, 가입일, 등록 과제 수, 계정 상태)
  • 기능:
    • 교사 신규 등록: 이름 및 이메일 입력 → 임시 비밀번호 발급 or 초대 메일 발송 (구현 방식 OI 연계)
    • 교사 계정 활성화 / 비활성화
    • 교사 상세 클릭 → SCR-ADMIN-02 이동
  • 정책: 관리자만 교사 계정을 직접 생성 가능. 교사 자율 회원가입 허용 여부 OI-03 연계 협의 필요

교사별 현황 조회

SCR-ADMIN-02 MVP포함
  • 표시 정보:
    • 해당 교사가 등록한 과제 목록 (과제명, 제출 기간, 등록 학생 수, 제출 현황)
    • 과제별 학생 제출 상태 요약 (제출 완료 / 미제출 / 분석 중)
  • 기능: 과제 클릭 시 해당 과제 상세(SCR-TCHR-03)로 이동 (읽기 전용)
  • 정책: 관리자는 조회만 가능. 과제 수정·삭제는 교사만 허용

3-3. 학생 화면 (STUDENT)

내 과제 목록

SCR-STU-01 MVP포함
  • 진입 방식: 과제 공유 링크 클릭 → 로그인/회원가입 → 과제 진입
  • 표시 정보: 과제명, 제출 기간, 내 상태(미제출 / 작성 중 / 제출 완료)
  • 기능: 과제 클릭 → 에디터 진입 (SCR-STU-02)

Verihum 에디터 (핵심 화면)

SCR-STU-02 MVP포함
  • 기능:
    • 텍스트 에디터 (최소 서식 지원: 굵게, 기울임, 줄바꿈)
    • 작성 과정 자동 기록 (백그라운드, 사용자에게 비노출)
    • 에디터 내 AI 어시스턴트 기능 (아래 별도 정의)
    • 제출 버튼 (제출 확인 모달 포함)
    • 임시저장 (OI-07 협의 필요)
  • 표시 정보: 과제명, 안내문, 작성 글자 수, 제출 기간 남은 시간
  • 정책:
    • 제출 후 수정 불가 (OI-08 협의 필요)
    • 과제 기간 종료 후 에디터 비활성화
    • 다른 탭/창 이동 시 처리 정책 협의 필요

에디터 내 AI 어시스턴트 기능

협의필요
  • 기능명: AI 도움 받기
  • 목적: 학생이 AI를 적극적으로 활용하도록 지원하되, 질문·검토·수정의 전 과정을 기록하여 '비판적 AI 활용도' 지표에 반영함. AI 사용 자체를 억제하는 것이 아니라, 비판적 사고 없이 AI 답변을 그대로 수용하는 행위를 구분하는 것이 목적.
  • 동작: 버튼 클릭 → AI 질문 직접 입력 → AI 제안 팝업 → 수락 / 수정 후 반영 / 거부 선택 → 모든 선택이 타임스탬프와 함께 기록됨
  • 평가 반영: AI 제안을 단순 수락한 비율보다, 수정·재해석 후 반영한 비율이 리포트의 '비판적 AI 활용도' 지표에 가산점으로 반영됨
  • 협의 필요: AI 어시스턴트 기능 MVP 포함 여부(OI-05), AI 제안 형태(OI-06), AI 공급사 선정, 학생 대상 고지 정책

제출 완료

SCR-STU-03 MVP포함
  • 표시 내용: 제출 완료 안내, 제출 일시
  • 버튼: 내 과제 목록으로 돌아가기

MVP 범위 정의

기능 범위는 MVP포함 / 협의필요 / 차기 3단계로 구분하며, 개발팀이 내용 검토후 기능범위 재정의 후 진행

기능 영역 세부 기능 MVP 범위 비고
인증 로그인 / 로그아웃 MVP포함
인증 이메일 회원가입 MVP포함
인증 소셜 로그인 (Google) 협의필요 구현 공수 vs 사용성 판단 필요
인증 이메일 인증 절차 차기 MVP에서 생략 가능
과제 과제 등록 MVP포함
과제 학생 링크 공유 방식 MVP포함 이메일 초대는 차기
과제 과제 수정 / 삭제 협의필요 제출 후 수정 정책 복잡 (OI-04)
과제 제출 현황 조회 MVP포함
에디터 Verihum 기본 에디터 MVP포함 핵심 기능
에디터 작성 로그 수집기 (Client) MVP포함
에디터 AI 어시스턴트 기능 협의필요 구현 복잡도 높음, OI-05 결정 시급
에디터 임시저장 협의필요 자동 vs 수동 저장 (OI-07)
리포트 개별 분석 리포트 열람 MVP포함
리포트 작성 과정 리플레이 협의필요 구현 방식 확정 후 공수 산정 필요 (OI-11)
리포트 학생 본인 결과 확인 차기 MVP는 교사 평가 도구에 집중
관리자 교사 목록 및 등록 (SCR-ADMIN-01) MVP포함 BN System 내부 운영자 전용
