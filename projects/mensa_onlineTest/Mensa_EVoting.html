4-2. 데이터 연동 구조

A. 쇼핑몰 DB → 전자투표 DB

[확인 필요] 쇼핑몰 측 API 제공 여부에 따라 아래 중 하나로 확정:

  • 방안 1 (권장): API 실시간 연동 — 티켓 구매 및 검표 즉시 투표 시스템에 반영. 투표 시작 이후 현장 구매자도 실시간 반영 가능.
  • 방안 2 (대안): 주기적 동기화 — 일정 주기(예: 1분 간격)로 쇼핑몰 DB 변경분(구매+검표) 동기화. 실시간성은 다소 떨어지나 구현 난이도 낮음.

⚠️ 쇼핑몰 DB에서 가져오는 데이터: ① 티켓 구매 정보(주문번호, 이름, 전화번호 등) ② 검표(출석) 여부 및 검표 시각. 두 가지 모두 동일한 연동 채널로 처리한다.

B. 위임 데이터 (구글 스프레드시트 → 전자투표 DB)

위임장은 별도 시스템에서 수집하며, 결과를 구글 스프레드시트로 정리한다. 전자투표 시스템은 해당 스프레드시트를 읽어와 위임 관계를 반영한다.
위임신청을 하였으나 위임자(수임자) 정보가 모두 공란인 경우, 해당 위임 건은 총회 등록 시 설정한 의장에게 자동 귀속된다.

  • 연동 방식: Google Sheets API를 통해 주기적 읽기 또는 관리자 수동 동기화 버튼
  • 스프레드시트 필수 컬럼: 위임인 이름, 위임인 연락처, 위임인 주문번호, 위임자(대리인) 이름, 위임자 연락처, 위임자 주문번호, 위임 등록 시각

💡 의장 설정 안내: 위임장에서 수임자(위임자) 정보가 모두 공란인 경우, 해당 위임 건은 위에서 설정한 의장에게 자동 귀속됩니다. 의장은 반드시 참석자(티켓구매자) 명단에 포함되어 있어야 하며, 등록 시 티켓구매자 DB와 매칭되지 않으면 저장이 불가합니다.

총회 상태 관리

총회에는 다음 3가지 상태가 있으며, 관리자가 수동으로 전환한다.

상태 설명 시스템 동작
준비중 (기본) 총회 생성 직후 상태. 안건 등록·수정 가능. 투표 불가. 사용자 접근 시 "총회가 아직 시작되지 않았습니다" 안내
투표 진행중 관리자가 수동 전환. 이 시점이 "투표 시작 시각"으로 기록되며, 위임 취소 판단의 기준이 된다. 사용자 본인인증 후 투표 화면 접근 가능. 안건별 열림/닫힘 제어 활성화.
종료 관리자가 수동 전환. 총회 전체 종료. 모든 안건 투표 불가, 결과 확정. 사용자 접근 시 "총회가 종료되었습니다" 안내.

💡 투표 시작 시각: 관리자가 총회 상태를 투표 진행중으로 전환한 시점이 시스템에 자동 기록된다. 이 시각이 위임 취소 자동 처리의 기준 시점이 된다. (예: 총회 14:00 시작, 관리자가 14:20에 "투표 진행중" 전환 → 14:20이 투표 시작 시각)

총회 목록

등록된 총회를 리스트로 표시한다. 각 총회를 클릭하면 해당 총회의 안건 관리 화면으로 진입한다.

5-3. 안건 관리

안건 등록/수정

필드명 필수 여부 설명
안건 번호 자동 해당 총회 내 순번 자동 부여 (1, 2, 3…), 순서 변경 가능
안건명 필수 투표 제목 (예: "멘사코리아 회원 연회비를 다음과 같이 조정한다.")
안건 설명 선택 부가 설명 텍스트
보기(선택지) 필수 (최소 2개) 단일선택 옵션, 개수 제한 없음, 순서 변경 가능
상태 기본값: 닫힘 토글로 열림/닫힘 자유 전환

안건 목록 화면 (총회 상세)

해당 총회에 등록된 안건을 리스트로 표시한다. 각 안건 우측에 상태 토글(열림/닫힘)이 있다.

안건 상태 설명 투표자 측 동작
닫힘 (기본) 투표 불가, 사용자 화면에 비활성 표시 안건이 보이되 선택 불가
열림 투표 가능, 관리자가 토글로 전환 선택지 선택 + 제출 가능

💡 관리자 자유 전환: 안건 상태는 닫힘 ↔ 열림 양방향 자유 전환이 가능하다. 실수로 닫아도 다시 열 수 있고, 투표 종료 후 재투표가 필요한 경우에도 재오픈할 수 있다. 안건을 닫으면 해당 시점부터 사용자 투표·수정이 불가하며, 다시 열면 투표·수정이 재개된다.

⚠️ 결과 집계 시점: 안건을 닫힘으로 전환하면 해당 시점 기준으로 결과가 집계·표시된다. 다시 열림으로 전환하면 결과가 변동될 수 있으므로, 최종 결과 확정은 총회 종료 상태에서 확인하는 것을 권장한다.

5-4. 결과 확인

안건별 투표 결과

안건 상태가 닫힘으로 전환되면 해당 안건의 결과를 집계하여 표시한다.

  • 표시 형식: 각 보기(선택지)별 득표수 + 비율(%) + 수평 막대 그래프(프로그레스 바)
  • 총 투표 수: 해당 안건에 투표한 총 인원 수 (본인 투표 + 위임 투표 합산)
  • 투표율: (투표한 인원 / 투표 가능 인원) × 100%

열림 상태에서도 실시간 중간 집계를 볼 수 있는지 여부 [확인 필요] — MVP에서는 닫힘 전환 후에만 결과 표시를 권장 (중간 집계 노출 시 투표 편향 우려)

안건별 참여자 명단

참여자 명단은 티켓구매자(투표권 원소유자) 기준으로 1행 1명 표시한다. 위임된 경우, 해당 행에 위임자 정보가 추가 노출된다.

표시 항목 설명 비고
티켓구매자 이름 투표권의 원래 소유자 항상 표시
티켓구매자 전화번호 티켓 구매 시 등록한 연락처 항상 표시
티켓구매자 주문번호 쇼핑몰 주문번호 항상 표시
투표 여부 투표함 / 미투표 항상 표시
투표 시각 최종 제출(또는 수정) 시각 투표한 경우에만 표시
위임 여부 본인 투표: "본인" / 위임 투표: "위임" 항상 표시
위임자 이름 실제 투표를 대리 행사한 사람 위임인 경우에만 표시, 본인 투표 시 "-"
위임자 전화번호 위임자의 연락처 위임인 경우에만 표시, 본인 투표 시 "-"

⚠️ 익명 투표 원칙: 참여자 명단에서는 "누가 투표했는지(또는 누가 대리했는지)"만 확인 가능. "어떤 보기를 선택했는지"는 관리자에게도 노출하지 않는다.

5-5. 출석 관리 (쇼핑몰 검표 데이터 연동)

현장 출석 확인은 쇼핑몰의 기존 검표 시스템으로 처리한다. QR 리더기로 티켓 QR을 스캔하면 쇼핑몰 측 검표 현황에 자동 반영(또는 관리자에 의해 수동 반영)되며, 전자투표 시스템은 쇼핑몰 DB에서 검표(출석) 여부를 가져와 투표권을 활성화한다.

연동 흐름

[현장 QR 리더기] ──스캔──▶ [쇼핑몰 검표 시스템] ──자동 반영──▶ [쇼핑몰 DB (검표 현황)]
                                                                        │
                                                                  연동 (API/DB)
                                                                        │
                                                                        ▼
                                                              [전자투표 시스템 DB]
                                                              출석 상태 반영 → 투표권 활성화

쇼핑몰 DB에서 가져올 검표 데이터

항목 설명 비고
주문번호 티켓 구매 주문번호 티켓구매자 매칭 키
구매자 이름 티켓구매자 이름
구매자 전화번호 티켓구매자 연락처
검표 여부 검표 완료 / 미검표 출석 확인 여부로 사용
검표 시각 QR 스캔이 처리된 시각 위임 취소 판단 기준으로 사용

⚠️ 검표 시각은 위임 취소 자동 처리의 판단 기준이 된다. 검표 시각이 총회 "투표 진행중" 전환 시각 이전이면 위임이 취소되고, 이후이면 위임이 유지된다. 쇼핑몰 DB에서 검표 시각 데이터를 제공하는지 [확인 필요]

동기화 방식

티켓 구매 데이터와 검표(출석) 데이터 모두 동일한 쇼핑몰 DB에서 가져오므로, 섹션 4-2-A의 쇼핑몰 DB 연동과 동일한 채널로 처리한다.

항목 내용
동기화 대상 ① 티켓 구매 정보 (신규 구매 반영) ② 검표 현황 (출석 상태 반영)
동기화 주기 쇼핑몰 연동 방식(API/주기적 동기화)에 따라 동일하게 적용
실시간 요건 투표 진행 중에도 현장 구매 + 검표가 실시간 반영되어야 하므로, 짧은 주기의 동기화 또는 실시간 API 필수

관리자 출석 현황 화면

관리자 페이지에서 출석(검표) 현황을 모니터링할 수 있다.

기능 설명
출석 현황 요약 전체 티켓구매자 수 / 출석(검표) 완료 수 / 미출석 수
출석자 명단 이름, 전화번호, 주문번호, 검표 시각
최종 동기화 시각 쇼핑몰 DB와 마지막으로 동기화된 시각 표시
수동 동기화 버튼 자동 주기 외에 관리자가 즉시 동기화를 실행할 수 있는 버튼

5-6. 위임 관리

구글 스프레드시트에 정리된 위임 데이터를 시스템에 반영한다.

  • 동기화: 관리자가 "위임 데이터 동기화" 버튼 클릭 시 스프레드시트에서 최신 데이터를 가져온다.
  • 위임 목록 표시: 위임인 이름 → 위임자(대리인) 이름 매핑 리스트를 관리자 화면에서 확인 가능
  • 위임 취소 처리 (자동): 위임인의 검표 시각이 총회 "투표 진행중" 전환 시각 이전인 경우, 해당 위임은 자동 취소되고 본인에게 투표권이 복귀된다.

💡 위임 취소 기준 정리:
- 기준 시각 = 관리자가 총회를 투표 진행중으로 전환한 시각 (시스템 자동 기록)
- 위임인의 검표 시각 < 기준 시각 → 위임 자동 취소, 본인 투표
- 위임인의 검표 시각 ≥ 기준 시각 → 위임 유지, 위임자가 투표
- 위임인이 미검표(미출석) → 위임 유지, 위임자가 투표

6-2. 투표 화면

화면 구성

상단 영역

  • 총회명 표시
  • 총 안건 수 표시 (예: "전체 5건")

안건 목록 영역

안건 번호 순서대로 세로 나열한다. 각 안건은 하나의 카드 형태로 표시된다.

안건 카드 요소 설명
안건 번호 예: "제1호 안건"
안건명 제목 텍스트
안건 설명 등록된 경우에만 표시
보기(선택지) 라디오 버튼 형태, 단일 선택
제출 버튼 안건별 개별 제출
상태 표시 하단 참고

안건별 상태에 따른 사용자 화면 동작

안건 상태 화면 동작
닫힘 + 미투표 안건 카드 표시되나, 선택지 비활성(disabled), "투표 준비 중" 표시
열림 + 미투표 선택지 활성, [제출] 버튼 표시
열림 + 투표완료 선택지 비활성(선택값 표시), [수정] 버튼 표시
열림 + 수정 중 [수정] 클릭 시 선택지 다시 활성, [제출] 버튼으로 전환
닫힘 + 투표완료 선택지 비활성 고정(선택값 표시), "투표가 마감되었습니다" 표시, 수정 불가
닫힘 → 다시 열림 관리자가 재오픈 시, 투표완료 사용자는 다시 [수정] 버튼 활성화

투표 제출 흐름

  1. 사용자가 선택지 하나를 선택한다.
  2. [제출] 버튼을 누른다.
  3. 확인 다이얼로그: "제출하시겠습니까?" → 확인 / 취소
  4. 제출 완료 시: 선택지 비활성 + 선택한 값 표시 + [수정] 버튼 노출
  5. 별도의 완료 화면이나 페이지 전환은 없다.

투표 수정 흐름

  1. 안건이 열림 상태인 동안 [수정] 버튼이 표시된다.
  2. [수정] 클릭 → 선택지가 다시 활성화된다.
  3. 다른 선택지를 고르고 [제출] → 기존 값을 덮어쓴다.
  4. 안건이 닫힘으로 전환되면 수정 불가. 다시 열림이 되면 수정 재개 가능.

6-4. 위임 취소 자동 처리

시나리오 처리
위임인의 검표 시각 < 총회 "투표 진행중" 전환 시각 위임 자동 취소 → 위임인 본인에게 투표권 복귀, 위임자 드롭다운에서 해당 위임인 제거
위임인의 검표 시각 ≥ 총회 "투표 진행중" 전환 시각 위임 유지 → 위임자가 해당 투표권 행사, 위임인 본인은 투표 불가
위임인이 미검표(미출석) 위임 유지 → 위임자가 해당 투표권 행사

💡 위임 취소는 총회 상태가 투표 진행중으로 전환되는 시점에 일괄 판정된다. 이 시점 이전의 검표(출석) 기록이 있는 위임인은 자동으로 위임이 취소되고 본인 투표로 전환된다.

9. 리스크 및 확인 필요 사항 종합

# 항목 리스크 수준 설명 담당
1 쇼핑몰 DB 연동 방식 High API 유무에 따라 전체 아키텍처 변경. 실시간 티켓 구매 + 검표 반영 요구사항 충족 여부 결정 개발팀 + 쇼핑몰 담당
2 쇼핑몰 검표 시각 데이터 제공 여부 High 검표 시각이 없으면 위임 취소 자동 판정이 불가. 대안(수동 판정 등) 필요 PM + 쇼핑몰 담당
3 안건 재오픈 시 결과 변동 Medium 관리자가 안건을 자유 전환할 수 있으므로, 닫힘 상태에서 확인한 결과가 재오픈 후 변동 가능. 최종 결과 확정 프로세스 합의 필요 PM + 고객
4 위임 대량 건수 UI/UX Medium 의장 등 특정인에 위임이 10건 이상 집중될 수 있음. 드롭다운 + 진행률 표시로 대응하나, 사용성 테스트 필요 디자인 + 개발팀
5 동시접속 규모 Medium 100~130명 예상. 서버 스펙 산정 및 부하 테스트 필요 PM + 개발팀
6 일정 (10일) High 4/15 기획 → 4/25 오픈. 개발·테스트 기간 극히 제한적. 범위 축소 or 리소스 증대 검토 PM
