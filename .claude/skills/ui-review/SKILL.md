# UI 리뷰 스킬

화면정의서, 와이어프레임, 디자인 시안을 사용성·일관성·접근성 관점에서 리뷰한다.

## Workflow

### Phase 1: 화면 파악
- 리뷰 대상 화면의 목적과 주요 사용자 흐름 파악
- 기획 의도(이 화면이 왜 필요한가) 확인
- 화면에 포함된 주요 요소 목록화 (GNB, 폼, 리스트, 버튼, 모달 등)

### Phase 2: 리뷰 수행
아래 5개 카테고리로 점검:

**사용성 (Usability)**
- 주요 태스크의 스텝 수는 적절한가
- CTA(Call to Action)는 명확한가
- 사용자가 현재 위치를 알 수 있는가
- 되돌리기/취소가 가능한가

**일관성 (Consistency)**
- 같은 기능이 같은 UI 패턴을 사용하는가
- 버튼 스타일, 색상, 위치가 통일되어 있는가
- 용어가 화면 간 일관되는가

**예외 화면 (Edge States)**
- 빈 상태(Empty State) 처리가 있는가
- 에러 상태 화면이 정의되어 있는가
- 로딩 상태가 고려되어 있는가

**접근성 (Accessibility)**
- 색상 대비는 충분한가
- 텍스트 크기는 적절한가
- 키보드/스크린리더 고려가 있는가

**반응형 (Responsive)**
- 모바일/태블릿 대응이 고려되어 있는가
- 컨텐츠 우선순위가 뷰포트별로 적절한가

### Phase 3: 출력

## Output Format

### 리뷰 요약
> 전체적인 평가 1~2문장

### 상세 피드백

| # | 화면/영역 | 카테고리 | 심각도 | 피드백 내용 | 제안 |
|---|----------|---------|--------|-----------|------|
| 1 |          | 사용성/일관성/예외/접근성/반응형 | Critical/Warning/Suggestion |  |  |

### 잘 된 점
- (긍정 피드백도 포함)

## Constraints
- 개인 취향이 아닌 사용성/일관성/접근성 근거 기반 피드백만 제공
- "~하면 좋겠다" 톤 유지, 지시형 톤 금지
- 기획 의도를 훼손하는 제안은 하지 않는다
