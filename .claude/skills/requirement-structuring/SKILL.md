# 요구사항 구조화 스킬

산발적이거나 모호한 고객 요구사항을 기능·정책·화면 단위로 분리하고 구조화한다.

## Workflow

### Phase 1: 수집·분석
- 입력된 요구사항 원문을 그대로 보존
- 각 요구사항에서 **기능(무엇을 하는가)** / **정책(어떤 조건·규칙인가)** / **화면(어디에 보이는가)**을 분리
- 중복·상충되는 요구사항 식별
- 암묵적 전제(명시되지 않았지만 당연히 필요한 것) 도출

### Phase 2: 구조화·판단
- 기능 목록 정리: 기능ID, 기능명, 설명, 관련 화면, 우선순위
- 정책 목록 정리: 정책ID, 정책명, 조건, 처리 방식, 예외
- MVP 기준으로 "지금 범위 / 차기 범위" 분류
- 모호한 지점에 "확인 필요" / "협의 필요" 태그 부여

### Phase 3: 출력
- 아래 형식으로 최종 결과물 출력

## Output Format

### 1. 요구사항 요약
> 원문 요구사항을 1~3문장으로 요약

### 2. 기능 목록

| 기능ID | 기능명 | 설명 | 관련 화면 | 우선순위 | 범위 | 비고 |
|--------|--------|------|----------|---------|------|------|
| F-001  |        |      |          | 상/중/하 | MVP/차기 |      |

### 3. 정책 목록

| 정책ID | 정책명 | 조건 | 처리 방식 | 예외 처리 | 비고 |
|--------|--------|------|----------|----------|------|
| P-001  |        |      |          |          |      |

### 4. 모호·리스크 포인트

| # | 항목 | 문제 유형 | 설명 | 상태 |
|---|------|----------|------|------|
| 1 |      | 모호/누락/상충/리스크 |      | 확인 필요 / 협의 필요 |

### 5. 확인 필요 사항
- 고객에게 확인할 항목
- 내부(개발/디자인)에 확인할 항목

## Constraints
- 원문에 없는 기능을 임의로 추가하지 않는다
- 불확실한 내용을 확정된 것처럼 기술하지 않는다
- 우선순위 판단 근거를 반드시 함께 제시한다
