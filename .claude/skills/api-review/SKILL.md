# API 리뷰 스킬

API 명세서와 백엔드 설계를 기획서·정책서 기준으로 정합성을 검토한다.

## Workflow

### Phase 1: 기획 정책 매핑
- 기획서에서 해당 기능의 비즈니스 로직·정책 추출
- 상태 전환도, 권한 매트릭스, 유효성 규칙 정리
- API가 커버해야 할 시나리오 목록화 (정상 + 예외)

### Phase 2: API 명세 대조
아래 관점으로 점검:

**기능 커버리지**
- 기획서 정의 기능이 API 엔드포인트에 모두 매핑되는가
- 누락된 API가 있는가

**비즈니스 로직 정합성**
- 기획서 정책(조건, 제한, 계산식)이 API 로직에 반영되었는가
- 상태 전환 규칙이 일치하는가
- 권한별 접근 제어가 기획과 일치하는가

**요청/응답 구조**
- 화면에 필요한 데이터가 응답에 포함되어 있는가
- 불필요한 데이터가 과도하게 내려오지 않는가
- 페이징, 정렬, 필터 등 목록 처리가 기획과 일치하는가

**에러 처리**
- 기획서 정의 예외 케이스에 대한 에러코드가 있는가
- 에러 메시지가 사용자에게 노출 가능한 수준인가
- 에러 상황별 화면 분기가 가능한 구조인가

### Phase 3: 출력

## Output Format

### 리뷰 요약
> 전체 API 설계 상태 1~2문장 요약

### 이슈 목록 (API별)

#### [API 엔드포인트 / 기능명]

| # | 항목 | 유형 | 심각도 | 기획 정의 | API 현황 | 비고 |
|---|------|------|--------|----------|---------|------|
| 1 |      | 누락/불일치/모호/리스크 | Critical/Warning/Minor |  |  |  |

### 개발자 확인 요청 사항
- API 설계 의도를 확인해야 할 항목 (질문 형태로)

### 기획 보완 필요 사항
- 기획서에서 정의가 부족하여 추가 정의가 필요한 항목

## Constraints
- 코드 리뷰가 아닌 **기획-API 정합성 검토**에 집중
- 개발자의 기술적 판단을 존중하되, 기획 의도와 다른 부분은 명확히 지적
- API 성능·아키텍처 의견은 "참고 의견"으로만 제시
