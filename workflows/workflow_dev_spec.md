# Workflow: 개발 전달용 스펙 문서 작성

## 언제 쓰나
기획이 확정된 후 개발팀에 넘길 스펙 문서가 필요할 때.

---

## Step 1 — 전제 확인

- [ ] 기능정의서 확정 여부
- [ ] 화면설계서 확정 여부
- [ ] 기술 스택 확인 (기본: PHP Laravel/CI + MySQL + Vanilla JS/jQuery)
- [ ] 외부 연동 API 여부 (결제, SMS, 소셜 로그인 등)

---

## Step 2 — 개발 스펙 문서 구조

```
1. 프로젝트 개요
   - 기술 스택 명시
   - 서버 환경 (Linux + Apache, AWS 등)
   - 개발 기간 / 담당자

2. DB 설계
   - ERD 또는 테이블 목록
   - 각 테이블: 필드명 / 타입 / 제약조건 / 설명
   - 인덱스 전략

3. API 명세 (백엔드 ↔ 프론트 통신)
   - 엔드포인트 / 메서드 / 요청 파라미터 / 응답 형식 / 에러 코드

4. 화면별 개발 요구사항
   - 화면 ID / 화면명
   - 기능 상세 (조건 분기, 유효성 검증 조건 수치로 명시)
   - 예외 케이스 처리 방법

5. 권한 관리 (RBAC)
   - 역할 정의
   - 역할별 기능 접근 권한 테이블

6. 외부 연동
   - 연동 서비스명 / 사용 API / 인증 방식

7. 비기능 요구사항
   - 성능: 페이지 로드 목표치
   - 보안: 인증 방식, 세션 관리, SQL Injection 대응
   - 브라우저 지원: Chrome 최신 / Edge / Safari (모바일 포함 여부)
```

---

## Step 3 — 작성 원칙

- 애매한 표현 절대 금지: "적절히" → "최소 8자, 최대 20자"
- 예외 케이스 빠짐없이: 권한 없음 / 데이터 없음 / 타임아웃
- DB 필드명은 실제 개발에서 그대로 쓸 수 있게 snake_case로
- 개발팀이 이 문서만 보고 착수할 수 있는 수준

---

## Step 4 — 산출물 저장

- 저장 위치: `outputs/개발스펙_프로젝트명_v1.0.docx`

---

## Step 5 — PM 보고

- "개발팀 전달 전 이 항목들은 확정이 필요합니다" 목록 제시
- 외부 연동 항목은 계정/키 준비 여부 확인 요청
