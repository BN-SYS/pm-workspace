# Workflow: 기능정의서 작성

## 언제 쓰나
PM이 "기능 정리해줘", "기능정의서 써줘", "범위 잡아줘" 라고 할 때.

---

## Step 1 — 요구사항 파악

PM에게 확인할 것:
- [ ] 클라이언트가 누구인지 (B2B / B2C / 공공기관)
- [ ] 프로젝트 성격 (신규개발 / 리뉴얼 / 고도화)
- [ ] 납기 및 우선순위
- [ ] 참고 사이트 또는 경쟁사 사례 유무

뻔한 건 묻지 말고 진행. 판단이 필요한 것만 올릴 것.

---

## Step 2 — 기능 목록 도출

1. PM이 말한 요구사항을 기능 단위로 분해
2. 빠진 기능 선제적으로 보완 (예: 로그인 → 비밀번호 찾기, 세션 관리, 권한 분리)
3. 기능 우선순위 분류:
   - **필수(Must)**: 없으면 서비스 불가
   - **권장(Should)**: 있어야 완성도 높음
   - **선택(Could)**: 추후 고도화 대상

---

## Step 3 — 기능정의서 구조 작성

```
1. 프로젝트 개요
   - 프로젝트명 / 클라이언트 / 작성일 / 버전
   - 배경 및 목적
   - 범위 (In-Scope / Out-of-Scope 명확히 구분)

2. 사용자 유형 정의
   - 역할별 권한 테이블 (RBAC)

3. 기능 목록
   - 대분류 > 중분류 > 기능명 > 설명 > 우선순위 > 비고

4. 기능 상세
   - 각 기능별: 설명 / 입력값 / 출력값 / 예외 처리 / 관련 화면

5. 비기능 요구사항
   - 성능, 보안, 브라우저 지원 범위 등
```

---

## Step 4 — 산출물 생성

- `tools/` 폴더에 기능정의서 생성 스크립트가 있으면 사용
- 없으면 Word 형식으로 직접 작성
- 저장 위치: `outputs/기능정의서_프로젝트명_v1.0.docx`

---

## Step 5 — PM 보고

보고 형식:
- "이 기능 목록으로 정리했고, [기능명] 부분은 범위 확정이 필요합니다"
- 클라이언트 확인이 필요한 항목은 별도로 표시

---

## 주의사항

- 애매한 표현 금지: "적절히", "필요시" → 수치와 조건으로 대체
- Out-of-Scope 명확히 선 그을 것 (나중에 범위 분쟁 방지)
- 개발팀이 보는 문서: DB 필드명, API 엔드포인트 힌트까지 포함
